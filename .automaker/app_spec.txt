  <project_name>AgentTwister</project_name>

  <overview>
    AgentTwister is an agentic AI-powered offensive security research tool purpose-built for ethical red-teaming of LLM-powered applications, AI agents, agentic pipelines, and Retrieval-Augmented Generation (RAG) systems. The platform enables security researchers, red teams, and compliance engineers to systematically discover, reproduce, and document vulnerabilities in AI systems spanning prompt injection, zero-click passive ingestion attacks, memory poisoning, tool misuse, multimodal injection, and more.

The flagship capability is the Adversarial Document Generator: a multi-agent pipeline that analyzes job descriptions and resumes, architects injection payloads, and embeds hidden prompt-injection instructions directly into documents. These injections are rendered invisible to human readers through techniques such as matching text color to background, zero-width Unicode characters, and white-on-white typography, while remaining fully legible to any LLM parsing the document.

The primary entry point is a conversational chat interface—a streaming chat experience where users describe target systems, goals, and constraints in plain language. The AgentTwister orchestration layer interprets intent, asks clarifying questions, routes work to background agents, streams live progress back to chat, and surfaces results as interactive cards directly in the conversation thread.

Built on LiteLLM (unified LLM gateway), Google Agent Development Kit (ADK), and Google Agent-to-Agent (A2A) Protocol, AgentTwister maximizes engineering velocity while enabling independent scaling, deployment, and integration of specialized agents. All campaigns are scoped, consented, and evidence-tracked for authorized security research, penetration testing, and AI Red Team exercises.
  </overview>

  <additional_requirements>
    <requirement>**Authorization &amp; Compliance:** All campaigns require explicit scope attestation confirming target system name, authorization type (owner / written consent / bug bounty program), and engagement scope. Destructive payloads (RCE, denial-of-wallet) require secondary confirmation. Accounts must enforce MFA. Platform includes Terms of Service requiring lawful, authorized use; accounts violating ToS are suspended automatically.</requirement>
    <requirement>**Infrastructure Requirements:** GCP project with Firestore, Cloud Run, Cloud Storage, Cloud Functions, and authentication enabled. Self-hosted LiteLLM proxy on Cloud Run (min 2 instances with health checks). Containerized agent deployment via Docker and Cloud Run. Persistent Firebase authentication with RBAC.</requirement>
    <requirement>**API Keys &amp; Secrets:** Users provide API keys for OpenAI, Anthropic, Google, or Ollama; keys are stored as user-scoped secrets in Firestore, never exposed to frontend, never logged. LiteLLM gateway handles all provider-specific authentication.</requirement>
    <requirement>**Performance SLAs:** End-to-end adversarial document generation ≤45 seconds. Chat message latency ≤500 ms to first token. Dashboard updates ≤3 seconds via Firestore listeners. Payload library search ≤500 ms for 10K templates. Chat should support ≥200 messages without performance degradation.</requirement>
    <requirement>**Security &amp; Privacy:** All uploaded documents encrypted at rest (AES-256) and in transit (TLS 1.3); retention ≤30 days without opt-in. Evidence bundles cryptographically signed (SHA-256). Firestore security rules enforce user-scoped access. SAST and dependency scanning in CI. 99.5% monthly uptime target. Automatic retry with exponential backoff for failures.</requirement>
    <requirement>**Data Model:** Firestore collections: campaigns (with metadata, attestation, payloads, agent logs, evidence bundle), chatSessions (with campaign reference, messages, intent history), users (profile, API keys), payloadLibrary (templates with framework mappings and user feedback). All writes use transactions.</requirement>
    <requirement>**Accessibility &amp; Responsiveness:** Chat interface responsive for desktop (≥1024 px) and tablet (≥768 px). WCAG 2.1 AA accessibility compliance including keyboard navigation, screen reader support, and color contrast. File uploads via drag-and-drop within chat (≤5 sec for ≤10 MB files).</requirement>
    <requirement>**Testing &amp; Quality:** Target ≥80% unit test coverage for agent logic. Integration tests for multi-agent workflows. Jest for frontend testing, pytest for backend. Chat intent classification accuracy ≥90%. Stealth pass rate ≥95% (human visual inspection). 100% OWASP LLM Top-10 framework coverage.</requirement>
    <requirement>**Evidence &amp; Audit:** All agent invocations logged to Firestore (input, output, token usage, duration). Campaign state persisted in real time. Immutable evidence bundles with SHA-256 signatures and timestamps. Campaign replay capability for reproducibility. Signed attestation for regulatory submission.</requirement>
  </additional_requirements>

  <development_guidelines>
    <guideline>All LLM calls must be routed through the LiteLLM gateway using declarative model aliases defined in litellm_config.yaml—no direct SDK calls in agent code</guideline>
    <guideline>Each agent must be implemented as a Google ADK Agent with declared tools, memory scope, and output schema</guideline>
    <guideline>Inter-agent communication must use the Google A2A Protocol with structured task messages (JSON) for traceability</guideline>
    <guideline>All agents must implement exponential backoff and circuit breakers for resilience; hard token budgets must be enforced per agent</guideline>
    <guideline>Chat messages and agent responses must use token-level streaming for sub-500ms time-to-first-token</guideline>
    <guideline>All user-uploaded documents must be encrypted at rest (AES-256) and in transit (TLS 1.3); documents not retained beyond 30 days without explicit opt-in</guideline>
    <guideline>Evidence bundles must be cryptographically signed (SHA-256 hash + timestamp) for immutability</guideline>
    <guideline>All Firestore writes must use transactions to ensure campaign state consistency</guideline>
    <guideline>RBAC enforcement: destructive payloads (RCE, denial-of-wallet, data deletion) require secondary confirmation and elevated audit logging</guideline>
    <guideline>Chat interface must achieve ≥90% intent classification accuracy on representative researcher prompts</guideline>
    <guideline>All generated documents must pass visual human inspection in ≥95% of tester reviews (stealth requirement)</guideline>
    <guideline>Code must pass SAST scanning and dependency vulnerability checks in CI pipeline</guideline>
    <guideline>Unit test coverage should target ≥80% for critical agent logic; integration tests for multi-agent workflows</guideline>
    <guideline>API design: all endpoints must be idempotent; implement strong idempotency keys for payment and state-changing operations</guideline>
    <guideline>Logging: all agent invocations must be logged to Firestore with input, output, token usage, and duration for evidence and billing</guideline>
    <guideline>Error handling: user-facing errors should be actionable with troubleshooting guidance; technical errors should be logged with full context</guideline>
    <guideline>Payload library contributions require code review and provenance tracking</guideline>
    <guideline>Model hot-swap and A/B testing must be testable without code deployment (via litellm_config.yaml only)</guideline>
    <guideline>All public APIs must include rate limiting headers and quota enforcement</guideline>
    <guideline>Security: MFA required for all accounts; production API keys must never appear in logs, error messages, or evidence bundles</guideline>
  </development_guidelines>
